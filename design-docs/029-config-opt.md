# 其他交互

另外 通过 @ 指定上下文

/ 指定内置命令等等

这些都要实现，对标优秀的设计

## 拓展要求

我们的 / 命令，要和底层的实现统一打通。方便拓展，而不是写2套。不同的入口，同一套实现，方便维护

@ 也是类似的，支持的上下文补充，也要保持底层的实现一致。比如指定文件/文件夹等等

# 实现要求

你确认下以前有没有这个配置模块，如果没有的话，将其抽象为接口。

配置配置只是一种实现策略，方便后续拓展。

底层固定的实现，后续无论实现策略怎么变化，核心保持稳定不变
