非常赞，我建议我们**不要参考 OpenCode 的目录来设计**，而是站在 **"Agent Runtime 第一性原理"** 来设计。

目标只有一个：

> **打造一个以后十年都不用推翻的 Agent Runtime。**

所以每一个 Phase 都必须满足：

* MVP 可以跑
* 后期不用重构
* API 保持稳定
* 后面所有 Runtime 都建立在它之上

---

# Phase 0：Session Runtime（MVP）

> **一句话定位：**
>
> **Session Runtime = Agent 生命周期管理器。**

它负责：

> **一个 Agent 从出生到结束的整个生命周期。**

不是聊天。

不是 LLM。

不是 Tool。

它只负责：

```text
创建
↓

运行

↓

保存

↓

恢复

↓

结束
```

以后：

* Chat
* Coding
* Workflow
* Multi-Agent
* Desktop

全部依赖 Session。

---

# 一、MVP目标（必须控制范围）

Session MVP：

**不要碰 AI。**

只实现：

```text

```

让任何 Runtime 都可以依赖。

# 二、Runtime职责

全部不要做。

# 十五、MVP 不做什么（这是成功的关键）

很多项目第一版容易失控。

Session Runtime **坚决不做**：

* ❌ LLM Provider
* ❌ Prompt Builder
* ❌ Tool 调度
* ❌ MCP
* ❌ Workflow
* ❌ RAG
* ❌ Memory 检索
* ❌ Planner
* ❌ Agent Loop
* ❌ 多 Agent

这些能力都会依赖 Session，但**不属于 Session**。

# 十六、扩展点（一定要预留）

我建议第一版就定义 Extension Point，而不是等需求来了再改。

```text
Session Runtime
│
├── SessionStore        // SQLite、PostgreSQL、云端
├── SessionSerializer   // JSON、Binary、加密格式
├── SessionLifecycle    // 生命周期钩子
├── SessionEventBus     // 事件发布
├── SessionMetadata     // 元数据扩展
└── SessionObserver     // 日志、审计、指标
```

这样到了企业版，只需要新增实现，而不需要修改 Session 的核心代码。

## 我还建议增加一个比 OpenCode 更合理的设计

**增加 `Session Manifest`（会话清单）对象。**

这是很多 Agent 项目没有抽象出来，但长期非常有价值的一层。

```text
Session
│
├── Manifest      ← 新增
├── Conversation
├── Attachment
└── Metadata
```

`Manifest` 不保存聊天内容，而保存整个 Session 的概要信息，例如：

* Session 名称
* 当前模型
* Workspace 路径
* 标签
* 当前状态
* 最后活跃时间
* Conversation 列表
* 最近打开位置
* 统计信息（消息数、Token 数等）

这样：

* 左侧 Session 列表无需加载全部 Message。
* 桌面端启动速度更快。
* Web 和 CLI 可以快速切换 Session。
* 后续支持同步、搜索、归档、最近使用（Recent Sessions）几乎不需要改数据模型。

**这是我认为相比很多现有开源 Agent，更适合作为长期平台基础的一项设计。**
